# Contributing

Thanks for improving SSH Tunnel. This project is a small native macOS app, so
changes should stay focused, tested, and easy to release.

## Before Opening A Pull Request

- Check that the change fits the app's scope: SSH config based menu bar tunnel
  management for macOS.
- For Swift code changes, write or update tests first, then implement the
  change.
- Prefer current Swift 6 APIs and avoid deprecated macOS APIs.
- Keep unrelated refactors out of feature or bug-fix PRs.
- Run the relevant checks locally:

```bash
swift test
make test
```

Use `make` or `make run` when the change affects the app bundle, icon handling,
or launch behavior.

## Commit And PR Title Style

Releases are generated by Release Please from Conventional Commit history. Use
the same style for commit messages and pull request titles:

```text
fix: resolve stale control socket cleanup
feat: add quick forward labels
docs: document Gatekeeper behavior
test: cover tunnel reconnect cap
chore: update release workflow
```

Use a scope when it makes the change clearer:

```text
fix(settings): preserve quick forward labels
feat(menu): add immediate health check action
```

Breaking changes must be explicit:

```text
feat!: change tunnel settings storage format
```

or include a footer:

```text
BREAKING CHANGE: tunnel settings now require migration before launch.
```

Release impact follows semantic-release semantics:

- `fix:` bumps patch.
- `feat:` bumps minor.
- `BREAKING CHANGE:` or `!` bumps major.
- `docs:`, `test:`, `chore:`, and similar maintenance changes normally do not
  create a release unless configured otherwise.

## Version Files

Do not manually bump release versions for normal feature or bug-fix work.
Release Please updates version files when it opens or lands a release PR.

Only touch version metadata when the PR is specifically about release/version
maintenance. In that case, keep these aligned:

- `Resources/Info.plist`
- `.release-please-manifest.json`
- `CHANGELOG.md`
- relevant version tests or docs

`CFBundleShortVersionString` is the user-visible semantic version in
`MAJOR.MINOR.PATCH` format without a leading `v`. `CFBundleVersion` is the
numeric macOS build version and must be monotonically incremented for
distributed builds.

## Test Expectations

- Swift behavior changes should include focused unit tests under
  `Tests/SSHTunnelTests`.
- UI or lifecycle changes should include the closest practical test coverage
  plus clear manual verification notes in the PR.
- Release workflow, Makefile, or metadata changes should include either tests
  or an explanation of why local verification is enough.

## Pull Request Checklist

Before marking a PR ready for review:

- The title follows Conventional Commit style.
- Swift changes were developed test-first where practical.
- Tests pass locally or the PR explains why they could not be run.
- User-facing behavior is documented in README, Help text, or release notes
  when needed.
- Version files are left to Release Please unless this is release/version
  maintenance.
- The PR description includes what changed, how it was tested, and any known
  follow-up.
